# 圣遗物预测器 - 打包方案

## 环境要求

- Python 3.12+
- uv (Python包管理器)
- Windows 10/11

## 依赖版本

```toml
[project]
dependencies = [
    "torch==2.5.1",      # PyTorch CPU版本
    "PyQt5==5.15.10",    # GUI框架
    "numpy==2.4.3",      # 数值计算
    "pandas==3.0.1",     # 数据处理
    "openpyxl==3.1.5",   # Excel读写
]
```

## 打包步骤

### 1. 创建虚拟环境并安装依赖

```bash
# 使用uv创建venv并安装依赖
uv sync
```

### 2. 安装打包工具

```bash
uv pip install pyinstaller
```

### 3. 执行打包

```bash
# 使用venv中的pyinstaller
.venv\Scripts\pyinstaller --clean build_working.spec
```

### 4. 打包完成

输出目录：`dist\圣遗物预测器\`

## 关键配置说明

### 本方案的特殊之处：使用 Tree 复制完整库目录

**核心思想**：不依赖 PyInstaller 的自动分析，而是使用 `Tree` 将整个库目录**原样复制**到打包目录。

```python
from PyInstaller.building.build_main import Tree

# Tree 的作用：将整个目录复制到打包输出目录
# prefix='numpy' 表示复制到 _internal/numpy/ 目录下
numpy_tree = Tree(get_numpy_path(), prefix='numpy', excludes=["*.pyc", "__pycache__", "*.lib"])
pandas_tree = Tree(get_pandas_path(), prefix='pandas', excludes=["*.pyc", "__pycache__", "*.lib"])
torch_tree = Tree(get_torch_path(), prefix='torch', excludes=["*.pyc", "__pycache__", "test", "testing", "*.lib", "*.dll", "*.pyd"])

# 将Tree添加到datas，PyInstaller会完整复制这些目录
a.datas += numpy_tree
a.datas += pandas_tree
a.datas += torch_tree
```

### 为什么要用 Tree 复制整个库？

| 问题 | 传统方式 | Tree方式 |
|------|---------|---------|
| PyInstaller分析遗漏 | ❌ 经常发生 | ✅ 完整复制，不会遗漏 |
| 动态导入失败 | ❌ torch._C等C扩展难处理 | ✅ 完整保留原始结构 |
| 子模块缺失 | ❌ pandas.core等可能丢失 | ✅ 整个包都在 |
| 编译后的.pyd文件 | ❌ 可能被忽略 | ✅ 完整复制 |

**代价**：体积稍大，但稳定性极高。

### 动态库的特殊处理

虽然用 Tree 复制了库目录，但 `.dll` 文件需要额外处理：

```python
# 1. 收集所有DLL到binaries列表（PyInstaller会放到_internal根目录）
binaries = []
binaries.extend(collect_dynamic_libs('numpy'))
binaries.extend(collect_dynamic_libs('pandas'))
binaries.extend(collect_dynamic_libs('torch'))

# 2. PyTorch 2.x 的核心DLL需要显式添加（collect_dynamic_libs收集不到）
for dll_file in ['torch_python.dll', 'torch.dll', 'torch_cpu.dll', 'torch_global_deps.dll']:
    dll_full_path = os.path.join(torch_path, 'lib', dll_file)
    if os.path.exists(dll_full_path):
        a.binaries.append((dll_file, dll_full_path, 'BINARY'))
```

**为什么DLL要单独处理？**
- Tree把DLL复制到 `torch/lib/` 子目录
- 但Windows加载DLL时优先搜索程序根目录
- `collect_dynamic_libs` 会把DLL放到 `_internal/` 根目录，确保能加载
- PyTorch 2.x 的 `torch_python.dll` 等核心DLL无法被自动收集，必须显式添加

## 常见问题

### Q: 打包时使用了错误的Python环境

**现象**：日志显示 `Python environment: D:\MiniConda`

**解决**：使用venv的pyinstaller
```bash
.venv\Scripts\pyinstaller --clean build_working.spec
```

### Q: 打包后提示缺少torch._C

**现象**：`cannot import name '_disabled_torch_function_impl' from 'torch._C'`

**解决**：确保spec文件中显式添加了torch核心DLLs

### Q: 训练功能无法使用

**现象**：训练时DLL初始化失败

**原因**：PyTorch 2.5.1使用OpenMP而非MKL

**解决**：确保 `torch/lib/libiomp5md.dll` 被正确打包（通过collect_dynamic_libs自动收集）

## 注意事项

1. **必须使用venv的pyinstaller**，避免使用系统或conda的pyinstaller
2. **不要修改torch DLLs的收集逻辑**，这是PyTorch 2.x打包的关键
3. **使用COLLECT模式**（文件夹形式），比单文件模式更稳定
4. **console=False** 关闭控制台窗口，发布版本使用
